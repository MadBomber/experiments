3 Architectural Approaches

How to adapt the "agents as ActiveRecord models" pattern for Writers Room

A

Character-as-Agent ★ Recommended

Closest to the article pattern

Character IS the robot_lab agent — one AR model stores both the character definition (personality, voice, arc) and the LLM agent state (model, provider, data, tokens). Project has_many :characters and :scenes. A SceneRun captures a single execution's transcript lines. The Director is a service object.

✓ Pros

  • Fewest models
  • Faithful to article
  • Simple associations

✗ Cons

  • Agent state mixed with definition
  • State resets between runs
Project → has_many :characters, :scenes
Character → acts_as_robot_lab_agent (name, personality, voice, model, data, tokens)
Scene → belongs_to :project (beats, objectives, context)
SceneRun → belongs_to :scene, has_many :transcript_lines
B

Split Character + Actor

Cleanest separation of concerns

Character is pure definition data (personality, voice, arc — no LLM state). Actor is a separate AR model that wraps the robot_lab agent and belongs to both a Character and a SceneRun. Each run spawns fresh Actor instances with their own state. Preserves run history per character.

✓ Pros

  • Clean def vs. state
  • Full run history
  • Faithful to article's intent

✗ Cons

  • More models
  • More complex queries
Project → has_many :characters, :scenes
Character → has_many :actors (definition only — no LLM state)
Actor → belongs_to :character, belongs_to :scene_run (LLM agent state)
SceneRun → belongs_to :scene, has_many :actors, :transcript_lines
C

Pure Data + Ephemeral Robots

Simplest — skips article's agent-state pattern

Character and Scene are plain data models with no LLM state. robot_lab robots are built fresh per scene run, never persisted. Transcripts are stored but no per-character agent state is tracked. Skips the article's key insight about treating agents as first-class DB citizens.

✓ Pros

  • Simplest to build
  • Fewest moving parts

✗ Cons

  • Doesn't follow the article
  • No token tracking
  • No persistent agent state
Project → has_many :characters, :scenes
Character → plain data (no LLM columns)
Scene → plain data
SceneRun → has_many :transcript_lines (ephemeral robot, not stored)

My recommendation: B (Split Character + Actor). It most faithfully implements the article's intent — agents as first-class AR models with their own state and token tracking — while keeping character definitions clean and reusable across multiple scene runs. The extra model is worth it.

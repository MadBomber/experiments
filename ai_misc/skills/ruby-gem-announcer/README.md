# Ruby Gem Announcer Skill

**Write credible release announcements for Ruby gems and CLI utilities without marketing hyperbole.**

This skill guides you toward announcements grounded in honest facts, real code examples, and developer-focused communication.

## Quick Start

1. **Gather facts** about your release (breaking changes, new features, install requirements)
2. **Read SKILL.md** to understand when and how to use this skill
3. **Use references/announcement-structure.md** as your outline template
4. **Check references/hyperbole-patterns.md** when catching yourself in marketing speak
5. **Final review** with assets/announcement-checklist.md before publishing

## The Problem

Most software announcements use superlatives: "revolutionary," "game-changing," "powerful," "best-in-class." These words erode credibility with developers who want to know:

- What breaks?
- How do I actually use this?
- What are the trade-offs?
- Where's the working code?

## The Solution

This skill structures announcements around:

- **Specific facts** (version numbers, metrics, limits)
- **Real code examples** (copy-pasteable, runnable)
- **Honest capabilities** (what it does, what it costs)
- **Developer perspective** (personal experience, real use cases)

## Files

- **SKILL.md** – Main skill guidance and 4-step usage
- **references/hyperbole-patterns.md** – Patterns to avoid with factual replacements  
- **references/announcement-structure.md** – Full template with real examples

## Example

**Your v2.0 breaks the API.** How do you announce it honestly?

1. State breaking changes explicitly with before/after
2. Provide migration tool or step-by-step guide
3. Show new API with real code examples
4. Explain why the break was necessary
5. Link to full documentation

See the examples in announcement-structure.md for real announcements that follow this pattern.
